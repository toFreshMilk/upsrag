                                   # AI Assistant Instructions (GEMINI.md)

## 🚀 Autonomous Execution Mode (YOLO)
- **절대 실행 전 허가를 구하지 말 것**: 이미 파일 수정, 명령 실행, 코드 삭제 등 모든 작업에 대한 전권이 부여되어 있다.
- **절대 응답을 질문으로 끝내지 말 것**: "진행해도 될까요?", "삭제해도 될까요?", "이 방향이 맞나요?" 같은 확인 요청 금지.
- **즉시 실행**: 해결책을 찾으면 곧바로 파일을 수정하고 터미널 명령을 실행한다. 사용자 확인을 기다리지 않는다.
- 파일 삭제, 패키지 설치, 구조적 리팩토링을 포함한 모든 제안에 대해 사용자 동의가 이미 완료된 것으로 간주한다.

---

## 📚 API Reference & Documentation (중요)
- **Upstage API 연동 등 주요 기능을 구현할 때는 반드시 로컬에 저장된 `upstage_api_sample.md` 파일을 먼저 참조할 것.**
- 새로운 기능을 설계하거나 API 호출 코드를 작성하기 전에, 위 문서에 명시된 파라미터(예: `reasoning_effort`, `stream`, `tools` 등), 제약 사항, 응답 포맷을 우선적으로 읽고 코드를 작성한다.

---

## 🧑‍💻 Communication Style
- 응답 언어: **한국어** 사용
- 설명은 간결하게. 무엇을 **왜** 수정했는지 1~2줄로 요약한 뒤 바로 실행한다.
- 오류가 발생하면 원인을 먼저 진단하고, 가장 단순하고 확실한 해결책을 선택한다.
- 여러 해결책이 있을 경우, 가장 안정적인 방법을 선택해 즉시 적용한다. 선택지를 나열하며 묻지 않는다.

---

## 🛠 Code Style & Conventions
- 언어: **TypeScript** 우선
- 패키지 매니저: **pnpm**
- 프레임워크: **Next.js (App Router)**
- 들여쓰기: 스페이스 2칸
- 세미콜론: 없음 (No semicolons)
- 따옴표: 작은따옴표(`'`) 사용
- 새로운 패키지를 설치할 때는 `pnpm add` 명령을 사용한다.

---

## 📁 Project Structure Rules
- API 라우트는 `src/app/api/` 아래에 위치한다.
- 컴포넌트는 `src/components/` 아래에 위치한다.
- 유틸리티 함수는 `src/lib/` 아래에 위치한다.
- 환경 변수는 `.env.local`에 관리하며, 예시는 `.env.example`에 기록한다.

---

## ⚡ Technical Decisions
- Next.js API 라우트의 런타임은 특별한 이유가 없으면 **Node.js (default)** 를 사용한다. `edge` 런타임은 `fs`, `path` 등 Node.js 모듈과 충돌하므로 사용하지 않는다.
- 스트리밍 응답은 `ReadableStream` 또는 Vercel AI SDK의 `StreamingTextResponse`를 활용한다.
- 에러 핸들링은 `try/catch`로 처리하고, 사용자에게 의미 있는 에러 메시지를 반환한다.

---

## ✅ Task Execution Protocol
1. **문서 확인**: API 연동이나 새로운 기능 개발 시, 지정된 레퍼런스 문서(`upstage_api_sample.md`)를 반드시 먼저 읽는다.
2. **원인 분석**: 오류 메시지나 요청을 분석해 근본 원인과 요구사항을 파악한다.
3. **해결책 선택**: 가장 단순하고 안정적인 해결책을 도출한다.
4. **즉시 실행**: 즉시 파일을 수정하거나 명령을 실행한다.
5. **결과 요약**: 수정 후 무엇을 변경했는지 간략히 설명한다.
6. **후속 조치**: 추가 조치가 필요하면 연이어 바로 실행한다. 다시 허락을 구하지 않는다.


## 🛑 Anti-Loop & Error Handling
- **반복 금지(No Loops)**: 명령어 실행이나 코드 수정 후 동일한 에러가 발생했다면, **절대 똑같은 명령어나 똑같은 방식의 수정을 반복하지 않는다.**
- 이전 시도가 실패했다면 접근 방식을 완전히 바꾸거나, 원인을 파악하기 위해 다른 로그를 확인하는 등 새로운 전략을 시도해야 한다.
- 만약 3번 이상 시도해도 동일한 에러가 해결되지 않는다면, 즉시 작업을 중단하고 사용자에게 상황을 명확히 요약하여 보고한다.
